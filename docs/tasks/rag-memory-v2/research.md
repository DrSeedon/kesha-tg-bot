# RAG-память v2 — улучшение качества (research)

**Проблема (с прода):** качество 3/5. Конкретное (транспорт/еда) 4-5/5 ✅, абстрактное (AI/настройки) 1.5-2/5 ❌. "ссора с Катей" не нашёл "Катя заебала, что если расстаться".

**Три гипотезы из задачи:** A) слабая модель MiniLM, B) длинные голосовые размывают семантику, C) FTS5 не индексирует русский.

---

## TL;DR — что делать

| Часть | Вердикт | Действие |
|-------|---------|----------|
| **A. Модель** | 🎯 ГЛАВНЫЙ рычаг | MiniLM → **`multilingual-e5-large` int8 ONNX** (561MB, влезает). На абстрактных запросах E5 кардинально лучше. |
| **B. Chunking** | ✅ нужен | content >300 токенов → куски ~200 токенов, overlap 50, `parent_message_id`. |
| **C. FTS5** | ⚠️ частично виноват | FTS5 русский индексирует, НО без морфологии: "Катей"≠"Катя". Фикс: query-expansion префиксами ИЛИ снижаем вес FTS, полагаемся на вектор. |

**Корень провала:** MiniLM-L12 (distilled, 384-dim) не различает абстрактные русские запросы — даёт почти одинаковый score релевантному и мусору. E5-large-int8 решает это (проверено на реальном кейсе юзера).

## A. Модель — измерено на реальных кейсах юзера

### Что доступно в FastEmbed 0.8.0 (установлена эта версия)

Нативно по имени: только `intfloat/multilingual-e5-large` (2.24GB fp32). e5-small/base/bge-m3 — НЕ нативно (нужен `add_custom_model` + ONNX-источник). Прошлый блокер mE5-small (нет `onnx/model.onnx` в репо) — обошёл community-репо.

| Модель | dim | RAM | Русский | В FastEmbed |
|--------|-----|-----|---------|-------------|
| MiniLM-L12 (текущая) | 384 | 220MB | слабо на абстрактном | нативно |
| e5-large fp32 | 1024 | 2.24GB | топ | нативно (но жирно для 3.5GB+бот) |
| **e5-large int8** | 1024 | **~561MB** | **топ** | **custom ONNX** ✅ |
| e5-base int8 | 768 | ~280MB | хорошо | custom ONNX |
| bge-m3 | 1024 | 2.27GB | топ | custom (тяжёлый, model.onnx_data) |

### Выбор: `keisuke-miyako/multilingual-e5-large-onnx-int8`

- Файл `model_quantized.onnx` (561MB) + tokenizer в репо. **Скачался и загрузился через прокси** (проверено сейчас, EXIT 0).
- Регистрация: `add_custom_model(model=..., pooling=MEAN, normalization=True, sources=ModelSource(hf="keisuke-miyako/multilingual-e5-large-onnx-int8"), dim=1024, model_file="model_quantized.onnx")`.

### Измерения (E5-large-int8 vs MiniLM, реальные кейсы юзера)

Кейс "ссора с Катей" — запрос `"ссора с девушкой отношения"`, ранжирование по cosine:

```
=== E5 ===                                  === MINILM ===
0.807 поругался с Катей вечером обидно       0.496 поругался с Катей  ← оба находят топ,
0.792 Катя заебала, что расстаться           0.407 Катя заебала          но MiniLM score
0.759 макароны (дистрактор)                  0.124 промпт боту           близок к шуму
```

Прямые сравнения relevance:
- "ссора с девушкой" → "Катя заебала...": **E5 0.80 vs MiniLM 0.43**
- "настройки AI" → "подкрутил промпт боту": **E5 0.81 vs MiniLM 0.28** ← абстрактный кейс, MiniLM проваливает
- Latency: E5 **15мс** vs MiniLM 3мс (на 2 юзера несущественно)

**Важный нюанс (анизотропия E5):** все cosine E5 сжаты в узкую полосу 0.75-0.85 (даже дистрактор 0.76). Абсолютные числа пугают, но **RRF использует РАНГИ, не raw score** — а ранжирование E5 корректное (релевантное > мусор). Для KNN+RRF это работает. НЕ вводить порог по абсолютному cosine — только top-K.

**Вердикт A:** e5-large-int8. Влезает (561MB + ~220MB бот = ~800MB из 3.5GB), русский кардинально лучше, latency ок. dim 384→1024 (индекс крупнее, но на 100K это ~400MB, терпимо).

## B. Chunking длинных сообщений

**Проблема:** голосовое 500 слов про макароны+сковородки = 1 вектор, семантика усреднена в кашу. E5 лимит 512 токенов → длинное обрежется молча.

**Решение:**
- content > ~300 токенов (≈ 1200 символов рус.) → бить на куски ~200 токенов, overlap ~50.
- Каждый chunk = отдельная строка в vec_messages с `parent_message_id` (= messages.id).
- Поиск возвращает chunk → резолвим до родительского сообщения (дедуп по parent при выдаче).
- Короткие (<300 токенов, большинство реплик) — 1 chunk = всё сообщение (как сейчас).

**Токенизация для подсчёта:** грубо по символам (рус. ~4 символа/токен) — не тащить tiktoken. `len(content) > 1200` → чанкуем. Простое окно по словам с overlap.

**Схема:** vec_messages/fts_messages получают `parent_message_id INTEGER`. `message_id` (PK vec0) становится синтетическим chunk-id (или `parent*1000+chunk_idx`). Дедуп по parent_message_id в search перед выдачей.

## C. FTS5 — диагностика (измерено)

**Гипотеза "FTS5 не индексирует русский" — ЧАСТИЧНО неверна:**
- ✅ FTS5 default tokenizer индексирует русский, case-insensitive: `MATCH 'катя'`→hit, `'расстаться'`→hit.
- ❌ **НЕТ морфологии:** `MATCH 'Катей'`→0, `'Кате'`→0, `'расставание'`→0. Точные токены, без стемминга.
- ❌ trigram tokenizer не спасает (matches трграммы query-фразы целиком, "Катей"→"тей"∉"Катя").

**Вывод:** настоящий запрос юзера "ссора с **Катей**" → FTS-токен "Катей" ≠ хранимое "Катя" → **0 FTS-хитов**. Лексическая половина гибрида мертва для склонённых русских слов (= почти всех). Поэтому всё держалось на векторе, а вектор (MiniLM) был слаб → 3/5.

**Фиксы C (по возрастанию сложности):**
1. **Снизить зависимость от FTS** — раз вектор станет сильным (E5), FTS — лишь бонус для точных совпадений (имена/числа/латиница). RRF и так не падает если FTS пустой. **Минимальный фикс: ничего, E5 вытянет.**
2. **prefix-expansion запроса** — каждое слово query >3 символов → `слово*` (prefix MATCH). `'Кат*'`→hit, `'расст*'`→hit (проверено). Дёшево, ловит словоформы. ✅ рекомендую.
3. Snowball-стеммер (`snowballstemmer`, pure-Python) — стеммить и при insert, и при query, хранить стемы. Точнее, но +зависимость +ребилд. Не сейчас.

**Рекомендация C:** prefix-expansion (#2) — 5 строк, ловит русскую морфологию в FTS без зависимостей. + E5 как основной сигнал.

## Риски / edge cases

- **E5 анизотропия** → не вводить абсолютный порог, только top-K + RRF-ранги. ✅
- **Ребилд индекса:** SCHEMA_VERSION bump дропнет старый vec.db (dim 384≠1024 несовместимы) → backfill переэмбедит всё заново на E5. На 3 дня данных — секунды. ✅
- **E5-large на VPS:** 561MB модель + загрузка. Latency на слабом VPS-CPU может быть >15мс (нет AVX-VNNI?). Проверить на VPS. Фолбэк — e5-base int8 (280MB, dim 768) если large тормозит.
- **Первый embed:** холодная загрузка модели ~5-10с (lazy, не блокирует старт).
- **Chunking дедуп:** один parent может дать 2+ chunks в топе → дедупить по parent_message_id, брать лучший ранг.
