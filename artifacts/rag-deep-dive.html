RAG-память Кеши: как это работает

Полная техническая карта: от нажатия кнопки «Отправить» в Telegram до семантического поиска по всей истории. С реальным SQL, диаграммами и интерактивными примерами.

1. Архитектура

Два потока данных: запись (каждое сообщение → embedding → в индекс) и поиск (запрос → гибридный поиск → результаты).

2. Запись сообщения (пошагово)

Пример: ты пишешь Кеше заказал пиццу пепперони.

Telegram → handlers.py
Aiogram получает update, создаёт PendingEntry, ставит в ChatState.
message_log.py → messages.db
INSERT INTO messages(chat_id, role, content, message_id)
VALUES(720740564, 'user', 'заказал пиццу пепперони', 23456);
Возвращает lastrowid = 1042.
Callback → asyncio.Queue
_enqueue_index(1042, 720740564, 'user', 'заказал пиццу пепперони') — через loop.call_soon_threadsafe. Не блокирует ответ юзеру.
_rag_worker → ThreadPoolExecutor
Фоновая asyncio-таска берёт из очереди, отправляет в выделенный поток: await rag.run(loop, "index_message", 1042, 720740564, 'user', 'заказал пиццу пепперони')

Добавляет E5-префикс: "passage: заказал пиццу пепперони"

[0.12, -0.45, 0.78, 0.03, ..., 0.33] — 384 числа

3. Поиск (пошагово)

Ты спрашиваешь: что я ел на прошлой неделе?

Кеша видит "на прошлой неделе"
→
search_memory(query=…)
→
run_in_executor
→
RagMemory.search()
Embedding запроса
"query: что ел на прошлой неделе" → MiniLM → query_vec (384 floats, ~25мс)
Векторный поиск (KNN)
SELECT message_id, distance
FROM vec_messages
WHERE chat_id = 720740564
  AND embedding MATCH :query_vec
ORDER BY distance LIMIT 20;
Находит по смыслу: "пиццу", "овсянка", "курицу купить" — потому что векторы близки.
FTS5 поиск (BM25)
SELECT message_id, rank
FROM fts_messages
WHERE fts_messages MATCH 'ел' AND chat_id = 720740564
ORDER BY rank LIMIT 20;
Находит по точному слову: только если "ел" буквально в тексте.
RRF Fusion (слияние результатов)
Reciprocal Rank Fusion с k=60 объединяет оба списка: 
score = 1/(60 + rank_vec) + 1/(60 + rank_fts)
Если сообщение в обоих списках — его score выше. Берём top-5.
JOIN с messages.db
SELECT m.content, m.timestamp, m.role
FROM msg.messages m
WHERE m.id IN (1042, 987, 1001, 955, 1033)
ORDER BY m.timestamp DESC;
Достаём полный текст и дату.
Результат → Кеше
[2026-06-25 19:30 | user] заказал пиццу пепперони
[2026-06-24 13:15 | user] на обед была овсянка с бананом
[2026-06-23 20:00 | assistant] Записал: ужин — курица с рисом, ~650 ккал
[2026-06-22 12:45 | user] купил курицу грудку 2 кг
[2026-06-21 21:10 | user] сырники на ужин сделал
Кеша вставляет это в ответ с цитатами и датами.
Как RRF работает — таблица рангов
СообщениеRank vecRank FTSRRF score
"заказал пиццу пепперони"1—1/61 = 0.0164
"овсянка с бананом"2—1/62 = 0.0161

"ел сырники" — найден и вектором (rank 5) и FTS (rank 1, слово "ел") → суммарный RRF score выше всех. Гибрид побеждает.

5. Почему один поток?

Проблема

Embedding занимает 15-40мс (CPU). Если делать в event loop — бот замрёт для ВСЕХ чатов на это время. А SQLite-коннект нельзя шарить между потоками.

Решение

Event Loop (async)
→
run_in_executor
→
ThreadPool(1)
  • max_workers=1 — один поток владеет SQLite conn + embedder. Нет race conditions
  • asyncio.Queue для индексации — запись в очередь мгновенная, не блокирует ответ
  • run_in_executor для поиска — event loop свободен пока поток работает
  • Embedder загружается лениво при первом вызове, в том же потоке

6. Backfill старых сообщений

При старте бота rag.backfill() индексирует всё что ещё не в vec.db:

SELECT id, chat_id, role, content FROM msg.messages
WHERE id NOT IN (SELECT message_id FROM indexed)
  AND role != 'system'
  AND trim(content) != ''
ORDER BY id;
  • Батчами по 64 сообщения — один embed-вызов на батч
  • Таблица indexed — идемпотентность: повторный запуск не дублирует
  • ~100K сообщений = несколько минут разово
  • Работает в фоне через asyncio.ensure_future(rag.run(loop, "backfill"))

7. Изоляция чатов

Два юзера — два изолированных мира. Поиск всегда фильтрует по chat_id.

chat_id: 720740564
Максим — свои сообщения, свои результаты поиска. Катины не видит.
chat_id: 893553748
Катя — свои сообщения, свои результаты. Максимовы не видит.
-- vec_messages создаётся с PARTITION KEY:
CREATE VIRTUAL TABLE vec_messages USING vec0(
    message_id INTEGER PRIMARY KEY,
    chat_id INTEGER PARTITION KEY,  -- шардинг!
    role TEXT,
    embedding FLOAT[384]
);
-- sqlite-vec физически шардит индекс по chat_id
-- Поиск WHERE chat_id=X сканирует ТОЛЬКО свой шард

8. Что если RAG сломался?

СценарийЧто происходитБот работает?
Модель не скачалась (HF недоступен)Embedder = None. search_memory делает fallback на LIKE '%query%' из messages.dbДа
vec.db corruptedДропаем vec.db + backfill из messages.db (source of truth)Да
OOM при загрузке моделиtry/except, embedder=None, fallback LIKEДа
sqlite-vec обновился, формат сломалсяDROP + CREATE + backfill. Индекс — производная, не source of truthДа
index_message упал на одном сообщенииЛогируется ошибка, пропускается. Сообщение останется в messages.db, проиндексируется при следующем backfillДа

Принцип: messages.db — единственный source of truth. vec.db — кеш/индекс, восстановим за минуты. Бот НИКОГДА не падает из-за RAG.
